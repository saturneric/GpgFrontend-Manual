GpgFrontend provides **Multi-Key Database Support**, allowing users to manage
multiple isolated OpenPGP key databases in one application. This is useful when
you want to separate keys for different projects, organizations, identities, or
security levels.

Starting with GpgFrontend v2.2.0, key database management is integrated into the
main **Settings** dialog. The old **GnuPG Controller** window has been removed.

## Features

- **Multiple Key Databases**: Manage separate key databases for different
  purposes or contexts.
- **Per-Database Engine Selection**: Choose the OpenPGP engine used by each key
  database, such as GnuPG or the experimental rPGP engine when available.
- **Flexible Switching**: Switch between configured databases from the main
  interface.
- **Customizable Management**: Add, edit, reorder, or remove key databases as
  needed.

## Benefits

- **Improved Separation**: Keep personal, project, organization, or test keys in
  different databases.
- **Better Workflow Control**: Use different key databases for different tasks
  without mixing unrelated keys.
- **Engine-Aware Testing**: Use GnuPG-backed databases for maximum compatibility
  and rPGP-backed databases for testing newer OpenPGP features such as OpenPGP
  v6.
- **Reduced Operational Risk**: Avoid accidentally using keys from the wrong
  project or security context.

## Accessing the Multi-Key Database Feature

To manage multiple key databases, follow these steps:

<Steps>

1. Open **Settings** from the main application menu.

2. Go to the **Key Database** section.

3. Here, you can view all configured key databases, including their names,
   paths, order, and selected OpenPGP engines.

4. Add, edit, reorder, or remove key databases as needed.

</Steps>

:::note

In GpgFrontend v2.2.0 and later, the previous **GnuPG Controller** window has
been removed. Key database management is now part of the unified **Settings**
interface.

:::

## Managing Key Databases

The **Key Database** settings page allows you to manage all configured key
databases.

### Adding a New Key Database

To add a new key database:

- Click **Add New Key Database**.
- Specify a display name for the database.
- Select the database path or use the default suggested location.
- Choose the OpenPGP engine when the option is available.

A new key database creates an isolated environment for generated or imported
keys.

:::note[Engine Selection]

GnuPG remains the recommended default engine for daily use and maximum
interoperability. The experimental rPGP engine can be used for supported
databases when testing newer OpenPGP features such as OpenPGP v6 or selected
post-quantum algorithms.

:::

**Relative Path Support:** If you enable the Relative Path option, the specified
absolute path will be converted to a path relative to the GpgFrontend
application directory. This is useful for Portable Mode or when moving the
application between different systems, because key database paths can remain
valid across different host environments.

### Editing and Reordering Key Databases

Right-click a database entry, or use the available management buttons, to access
actions such as:

- **Move Up / Move Down**: Reorder the database list.
- **Move to Top**: Move a database to the top of the list. The first database is
  treated as the default key database.
- **Edit**: Rename the database, adjust its path, or change supported settings.
- **Remove**: Remove the database from the GpgFrontend configuration.

:::caution

Some changes to key database settings may require restarting GpgFrontend before
they take full effect.

:::

### Switching Between Databases

- Use the **Key Toolbox** dropdown in the main interface to switch between
  configured databases.
- Select the desired database, and the corresponding keys will be displayed.
- Operations such as key generation, import, export, encryption, signing, and
  verification will use the currently selected key database where applicable.

### Changing the Engine of a Key Database

Each key database is bound to one engine at a time, and this includes the
**default key database** (the first entry in the list, also called the default
channel). The engine you select determines which key store GpgFrontend reads at
that database's path. A GnuPG keyring and an rPGP `gf_keydb.sqlite` store can sit
in the same directory, but they are **separate stores and are never merged**.

This matters most for the default key database, which normally points at your
GnuPG home directory:

- If you set the default database's engine to **rPGP** and restart GpgFrontend,
  the application creates and reads a `gf_keydb.sqlite` file inside that GnuPG
  home directory and shows only the rPGP keys. Your existing **GnuPG keys are
  not displayed** in the default database while rPGP is active.
- If you switch the default database's engine back to **GnuPG** and restart, the
  application reads the GnuPG keyring again, and the **rPGP store is not read**,
  so its keys are not displayed.

:::caution[Switching engines hides keys, it does not delete them]

Changing a database's engine does not move, convert, or delete any keys. Keys
that belong to the inactive engine simply are not shown, because GpgFrontend only
reads the store for the currently selected engine. To see a given set of keys
again, select the engine that owns them and restart if prompted.

If you need both GnuPG and rPGP keys available at the same time, create
**separate key databases**, one per engine, instead of switching the engine on a
single database.

:::

## About `gf_keydb.sqlite`

Starting with the 2.2 series, GpgFrontend may create a `gf_keydb.sqlite` file in
configured key database directories.

For **rPGP** key databases, this SQLite database stores rPGP-managed key material
and metadata.

For **GnuPG** key databases, private and public key material remains managed by
GnuPG. In this case, `gf_keydb.sqlite` is used only for GpgFrontend-managed
metadata and may be empty.

:::tip[How each engine stores keys]

The two engines keep your keys in fundamentally different places, which affects
backups and security expectations. For a full explanation, see
[How Keys Are Stored (GnuPG vs rPGP)](/advanced/key-storage).

:::

:::caution[Backup Notice]

Back up each key database according to its engine type.

For GnuPG databases, back up the corresponding GnuPG home directory and your
revocation certificates.

For rPGP databases, back up the database directory that contains
`gf_keydb.sqlite`, because it may contain rPGP-managed key material.

:::

## Use Cases for Multi-Key Databases

- **Project Isolation**: Keep keys for different projects separate to avoid
  accidental cross-use.
- **Personal and Work Separation**: Store personal and work-related keys in
  different databases.
- **Organization or Team Separation**: Maintain separate databases for different
  departments, teams, or customer environments.
- **Testing New Engines**: Use a dedicated rPGP database to test OpenPGP v6 or
  experimental algorithms without affecting your main GnuPG keyring.
- **Sensitive Key Handling**: Keep high-value or sensitive keys in a dedicated
  database with stricter backup and access procedures.

## Tips for Effective Multi-Key Database Usage

- **Use clear names**: Name databases according to their purpose, such as
  `Personal`, `Work`, `Project Alpha`, or `rPGP Test`.
- **Check the active database before operating**: Make sure you are using the
  intended database before importing keys, generating keys, encrypting files, or
  signing data.
- **Back up regularly**: Back up each database and store backups in secure,
  encrypted locations.
- **Separate stable and experimental workflows**: Use GnuPG-backed databases for
  daily work and separate rPGP-backed databases for testing experimental
  features.
- **Review database contents periodically**: Remove unused keys, check
  expiration dates, and verify that keys are stored in the intended database.
